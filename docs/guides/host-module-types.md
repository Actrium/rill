# Typing `host:*` modules in a guest project

A rill guest imports host capabilities as ambient modules:

```ts
import { getText, putText } from 'host:store'
import { fetch } from 'host:net'
```

`host:*` is not an npm package — it is a capability the host injects at runtime.
This guide is the supported convention for giving these imports **types**, so guest
authors get editor completion and `tsc` passes with **no** `@ts-expect-error`.

## Types are editor/typecheck-only — they never affect the build

The rill CLI (`cli/build.ts`) resolves `host:*` imports itself at bundle time and
leaves them as the runtime capability calls. That resolution is **independent of TypeScript
types**: a guest builds and runs correctly whether or not a `.d.ts` is present. So typing is
purely a developer-experience concern — nothing here changes how a guest is compiled or shipped.

## The mechanism: ambient `declare module 'host:*'`

Ambient module declarations are the **officially supported** way to type `host:*`. A host
publishes a single declaration file — conventionally `host-modules.d.ts` — that declares one
ambient module per capability it registers:

```ts
// host-modules.d.ts  (generated by the host from its capability descriptors)
declare module 'host:store' {
  export function getText(key: string): Promise<string | null>
  export function putText(key: string, value: string): Promise<void>
}
declare module 'host:net' {
  export function fetch(url: string, init?: RequestInit): Promise<Response>
}
```

A host may generate this file from its own rpc / subscription descriptors; rill only fixes
**where the file lives** and **how the compiler finds it** (below). rill does not prescribe how
the host generates it.

## Binary fields map to `Uint8Array`

A capability may declare that a request or response field is a **byte stream** rather
than JSON-encoded data (see `binary` on `rpc()` in `src/contract/index.ts`). rill fixes
the one **mapping rule** the host's generator MUST honour so a declared byte field types
as bytes on both ends:

> For a capability descriptor, a field name listed in `binary.input` maps to a
> **`Uint8Array`** parameter/field in the emitted signature, and a field name in
> `binary.output` maps to a **`Uint8Array`** in the emitted return type. Every other
> field keeps its JSON-derived type. A descriptor with no `binary` declaration
> generates exactly as before — no `Uint8Array` appears.

Worked example — the `host:store` byte capabilities

```ts
// descriptors
putBytes: rpc<{ key: string; value: Uint8Array }, { version: number }>({ binary: { input: ['value'] } })
getBytes: rpc<{ key: string }, { value: Uint8Array; version: number } | null>({ binary: { output: ['value'] } })
```

generate

```ts
declare module 'host:store' {
  export function putBytes(key: string, value: Uint8Array): Promise<{ version: number }>
  export function getBytes(key: string): Promise<{ value: Uint8Array; version: number } | null>
}
```

The manifest's `binaryCapabilities` list (emitted only when non-empty) names exactly the
capabilities carrying such a mapping, so the generator — and any publish gate — knows which
descriptors to apply the rule to. On the wire these fields cross as raw bytes (the RBS1
envelope, or the QuickJS `$b64` sidecar), never as a JSON number-array; the type simply
reflects that. See [`../../contracts/store-net-bytes.CONFORMANCE.md`](../../contracts/store-net-bytes.CONFORMANCE.md).

## The convention: where it goes, how it is picked up

1. Place the host-provided declaration file at the **guest project root** as
   `host-modules.d.ts`.
2. Have TypeScript pick it up by **one** of these (all equivalent — pick per your `tsconfig`):
   - it falls under the project's `include` globs (the default for a root-level `*.d.ts`), or
   - add it to `compilerOptions.types` if your `types` array is otherwise restricted, or
   - a `/// <reference path="./host-modules.d.ts" />` in an entry file.

Because it is an **ambient** declaration, no runtime import of the file is needed — its presence
in the compilation is enough for `import ... from 'host:store'` to type-check.

## Stable coupling to `rill/guest`

A generated `host-modules.d.ts` may reference rill's own guest types, e.g.:

```ts
import type { RillKeyEvent } from 'rill/guest'
```

rill keeps the **`rill/guest` type export surface stable** so host-generated declarations can
depend on it without breakage. Treat `rill/guest` as the single stable source of shared guest
types referenced from `host:*` declarations.

## Done when

A first-party guest app that removes its `@ts-expect-error` comments above `host:*` imports
type-checks cleanly (`tsc --noEmit` passes) with only the host's `host-modules.d.ts` added at
the project root — no build-config change.
